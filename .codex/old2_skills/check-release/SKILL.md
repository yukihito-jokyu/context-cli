# リリースチェック

修正せず、変更全体がリリース可能かを証跡に基づいて判定する。これをフロー最後の人間ゲートとする。

## 必須条件

- PRDの3桁IDを受け取る。
- 実装レビューでCriticalとHighが解消済みである。
- 必要な現在Architecture更新が完了している。
- コード、テスト、成果物をこのフェーズでは修正しない。

## 1. 対象を特定する

PRDに対応するGit差分全体を対象にする。既存の無関係な差分を区別できなければ停止する。

`prd.md`、`implementation-plan.md`、Engineering Foundation、関連ADR、現在Architecture、コード、テスト、設定、依存、品質ゲート結果を読む。

## 2. 必要な検証を再実行する

最終検証後に差分が変わった、証跡が不足する、または高リスク領域を変更した場合だけ関連テストと品質ゲートを再実行する。実行不能な検証は理由、代替確認、影響を明示し、重大な証跡不足はBlockedとする。

## 3. 判定する

次を`Pass / Blocked / Not applicable`で確認する。

1. Goal、Success Metrics、全ACを実装とテストで確認できる
2. 制約、NFR、ADR、Engineering Foundationに違反しない
3. 必要なテスト、Lint、型検査、ビルド、セキュリティ検査が成功する
4. 設定、データ、移行、権限、外部サービスへの影響が確認済みである
5. 監視、ログ、復旧、Rollbackがリスクに見合う
6. 現在Architectureが実装と一致する
7. 新規依存、外部コード、ライセンスを確認済みである
8. セキュリティ、プライバシー、アクセシビリティ、法令・契約上の阻害がない
9. 未解決事項がリリース阻害またはFollow-upへ分類済みである

1つでも未解決のBlockedがあれば総合結果はBlockedとする。問題は責務を持つフェーズへ戻し、修正後に差分全体を再確認する。

## 4. 人間が最終確定する

各項目の結果、根拠、未実行項目、残存リスク、Follow-up、総合結果候補を提示する。判断が必要な項目は一度に1つずつ質問する。

人間の明示承認後だけリリース可能と確定する。独立したリリースチェック文書やPRDへの詳細履歴は作らない。必要な証跡はテスト結果、CI、Git差分、PR説明へ残す。

## 5. 引き渡す

Pass時は次を提示する。

- リリース後に確認する主要ACとSuccess Metrics
- 監視、ログ、エラー、移行結果の確認項目
- 切り戻し・復旧条件
- Follow-upの記録先

このSkillはコミット、PR作成、push、リリース、デプロイ、本番操作をしない。
