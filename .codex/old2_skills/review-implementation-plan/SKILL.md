# 実装計画レビュー

設計レビューとTask・テスト計画レビューを一度に行い、実装開始前の最後の人間ゲートとする。

## 必須条件

- PRDの3桁IDまたは具体的な`implementation-plan.md`パスを受け取る。
- 対象`prd.md`が`Accepted`である。
- 関連ADRが存在する場合は計画と一緒にレビューする。

## 1. 独立して読み直す

PRD、計画、Engineering Foundation、関連ADR、現在Architecture、コード、テスト、設定、Git差分を読む。作成時の推論や自己評価は根拠にしない。

## 2. 統合レビューする

- 全ACに実装・テストの対応がある
- 設計が既存の責務境界と依存方向に適合する
- 公開API、データ、互換性、移行、設定への影響が明確である
- セキュリティ、並行性、失敗時整合性、復旧が扱われる
- Work Unitが独立して検証可能で、依存順が妥当である
- Files to Changeが実現可能で、過剰な先行固定がない
- テストが正常系、主要失敗系、境界条件、回帰を覆う
- 品質ゲートと未実行時の扱いが明確である
- ADR化すべき重要判断が計画内に埋没していない
- 不要な文書更新や過剰設計がない

指摘は`Blocking / Major / Minor`に分類し、一覧提示後、判断が必要なものを一問ずつ確認する。

## 3. 修正する

合意した修正を`implementation-plan.md`と、同時レビュー対象のADRへ反映する。PRDのScopeやAC変更が必要なら自動修正せず`define-prd`へ戻す。

新規の重要判断が見つかった場合は、このフェーズ内で選択肢と推奨案を確認し、必要なADRを作成・レビューする。別の候補一覧文書は作らない。

## 4. 確定する

Blockingがなく、主要リスクと検証方法が明確で、連続実装可能な場合だけ`Accepted`を提案する。人間の明示承認後に計画と必要なADRのStatusを確定する。

レビュー履歴ファイルは作らない。`Status Reason`には主要な修正と受容リスクだけを書く。

完了時は指摘、修正、残存リスク、次フェーズの`implement-plan`を報告する。
