# 実装レビュー

実装差分全体を独立して確認し、重大な問題を解消してリリースチェックへ渡す。

## 必須条件

- PRDの3桁IDを受け取る。
- 実装時の非公開推論や自己評価を根拠にしない。
- 同一セッションの場合は実装後に正本、差分、テスト結果を読み直し、独立したレビュー状態を作る。

## 1. 差分を特定する

Git履歴、作業ツリー、実装開始前の状態、実装完了報告から対象差分を特定する。既存の無関係な差分を除外し、分離できなければ基準を確認する。

## 2. 正本を読む

`prd.md`、`implementation-plan.md`、Engineering Foundation、関連ADR、現在Architecture、対象コード、テスト、設定、品質ゲート結果を読む。

## 3. レビューする

- 全ACと制約を満たす
- 計画、ADR、依存方向に適合する
- Scope外や不要な変更がない
- API、互換性、データ、設定、移行が妥当である
- エラー処理、境界条件、並行性、復旧が妥当である
- セキュリティとプライバシー境界が守られる
- テストが主要経路、失敗経路、回帰を検証する
- 新規依存、生成物、ライセンス、運用影響に問題がない

指摘を`Critical / High / Medium / Low`で重要度順に一覧提示する。各指摘にファイル・行、根拠、影響、推奨修正を付ける。

## 4. 修正と再検証を行う

`Critical / High`と判断が必要な`Medium`を一問ずつ確認する。人間が採用した指摘はコードとテストを修正し、対象テスト、回帰テスト、品質ゲートを再実行する。

Scope、AC、重要設計、ADRの変更が必要なら直接修正せず、PRDまたは実装計画レビューへ戻す。レビュー報告書やTask別レビュー履歴は作らない。

## 5. 完了候補を提示する

次を満たす場合だけ`Approved`候補を提示する。

- CriticalとHighが残っていない
- 採用したMediumが解消済み
- 未対応項目がFollow-upまたは受容リスクに分類済み
- 必要な検証が成功し、未実行理由が許容可能
- 修正後の差分全体を再レビュー済み

人間の明示確認を最終リリース判断とは扱わない。結果は会話とGit差分を証跡とし、PRD文書へレビュー履歴を追記しない。

完了時は指摘、修正、検証、残存リスク、次フェーズの`update-current-architecture`を報告する。
