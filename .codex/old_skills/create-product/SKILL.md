# Product作成

入力されたテーマを `$grill-me` で深掘りし、人間が確定した内容だけから `docs/product.md` のDraftを生成する。

## 必須条件

- 新規作成ではプロダクト構想、更新では変更したい方針を一文以上のテーマとして受け取る。
- テーマが曖昧または未指定でも、最初に `$grill-me` を使ってテーマを確定する。
- `$grill-me` が利用できない場合は代替質問を行わず、必要なスキルがないことを伝えて停止する。
- 出力には日本語を使う。
- 出力先は `docs/product.md` に固定する。ユーザーが明示した場合だけ別の出力先を使う。
- `docs/ai-driven-development/templates/product.md` を必ず使用する。存在しない、または構成が不正な場合は独自構成で生成せず、問題を報告して停止する。

## 1. 既存資料を調査する

議論の前に次を探して読む。

1. `docs/ai-driven-development/templates/product.md`
2. 既存の `docs/product.md`
3. テーマに関連するAccepted成果物

資料から判明する内容を人間へ質問しない。既存プロダクトの事実確認が必要な場合だけコードを調査する。

実行時は `docs/AI-driven-development.md` を参照しない。本スキルの記載規則に従う。

情報が矛盾する場合は、既存の `docs/product.md` を現在のプロダクト意図の正本として扱う。ただし、今回の議論で人間が明示的に変更を確定した内容は上書き根拠とする。その他の矛盾は自動解消しない。

## 2. grill-meで議論する

必ず `$grill-me` を使用し、質問を一度に1つずつ行う。AIが事実や方針を推測で補完してはならない。

Draft生成前に、少なくとも次を人間と確定する。

- Product Vision
- Target Users
- Core User Problems
- Value Proposition
- Product Goals
- Current Focus
- Product Principles
- Constraints

未確定事項は `Open Questions`、人間が暫定的に許容した前提だけは `Assumptions` として整理する。中核項目が未確定なら議論を継続する。Success Metricsなど中核項目以外は、未確定であることを `Open Questions` に明記すればDraft生成を許可する。

既存ファイルの更新では、テーマに関係する箇所だけを議論する。既存資料との矛盾を発見した場合は、自動解消せず人間の判断を得る。

## 3. 生成承認を得る

中核項目の確定内容と残る `Open Questions` を要約し、「議論を終了してDraftを生成するか」を人間へ確認する。明示的な承認を得るまでファイルを書き換えない。

議論履歴を別ファイルへ保存しない。重要な判断履歴が必要な場合だけ、生成後にADR候補として提示する。

## 4. Draftを生成する

テンプレートの見出し構成と順序を維持する。該当しない項目も削除せず `Not applicable` と記載する。テーマ固有の補足は既存項目内へ記述し、独自見出しは人間が承認した場合だけ追加する。

Statusは常に `Draft` とする。既存のStatusが `Accepted` または `Implemented` でも `Draft` へ戻し、`Status Reason` に変更テーマと再レビューが必要な理由を書く。`Accepted` への変更は行わない。

更新時はテーマに関係する箇所だけを変更する。整理だけを目的として既存内容を削除、言い換え、再構成しない。実装内容を単純に転記しない。

本文へ出典欄を追加しない。根拠は調査した資料と今回の議論で人間が確定した内容に限定する。

## 5. 自己検証する

生成後に次だけを確認し、必要なら修正する。

- テンプレートの見出しと順序を維持している
- Statusが `Draft` である
- 未確定事項と暫定的な前提を正しく分類している
- テーマ外の既存内容を意図せず変更していない
- 入力資料または人間が確定した内容と矛盾していない

内容の妥当性レビューは行わない。別の `document-review` に委ねる。

## 6. 次工程を提示する

複数のPRD候補がある場合は候補だけを完了報告へ提示し、`product.md` へ詳細計画を書かない。

更新により既存PRDへ矛盾または影響が生じても自動更新しない。影響を受ける成果物と矛盾内容を一覧にし、`document-review` 後に対応する別スキルへ委ねる。

完了時は次を簡潔に報告する。

- 作成または更新したファイル
- 変更したセクション
- 残る `Open Questions`
- 影響を受ける既存成果物
- ADR候補
- PRD候補
- 参照した資料
- 次工程として `document-review` が必要であること
