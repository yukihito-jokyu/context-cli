# PRD作成

同一セッションの `$grill-me` で確定した内容だけを使い、1つのユーザー課題を定義する `prd.md` のDraftを生成する。

## 必須条件

- 新規作成では解決したいユーザー課題、更新では変更したいPRD方針を入力として扱う。
- PRD境界が同一セッションで議論されていない場合は、生成前に必ず `$grill-me` を呼び出す。
- テーマまたはPRD境界が曖昧でも、独自の簡易質問フローで代替せず `$grill-me` を使う。
- `$grill-me` が利用できない場合は、必要であることを明示して停止する。
- 対話と本文には原則として日本語を使う。識別子、パス、既存文書の表記は維持する。
- 実行時は `docs/AI-driven-development.md` を参照しない。本スキルの記載規則に従う。
- 新規作成では `docs/product.md` と `docs/engineering/*.md` が存在し、原則として `Accepted` でなければ停止する。既存PRDの更新でも、参照先が存在しない、未Accepted、または矛盾している場合は推測で進めない。

## 1. 既存資料を調査する

次の順で、存在する資料を読む。

1. `docs/ai-driven-development/templates/prd.md`
2. `docs/product.md`
3. `docs/engineering/technology.md`
4. `docs/engineering/structure.md`
5. `docs/engineering/development-rules.md`
6. 関連するAccepted済みPRDとADR
7. 更新時は対象の既存 `prd.md`

テンプレートが存在しない、見出し構成に不足または不整合がある場合は、独自構成で生成せず問題を報告して停止する。テンプレート自体は修正しない。

資料から判明する内容を質問しない。ユーザー課題または現行挙動の事実確認が必要な場合だけコード、設定、テスト、Issueを調査する。コードに存在する挙動を、意図した要求だと推測してはならない。

既存 `prd.md` を現在のPRD意図の正本として優先する。ただし、今回の `$grill-me` で人間が明示的に変更を確定した内容は上書き根拠にする。資料間の矛盾は自動解消せず、矛盾箇所と影響を示して人間に正本を確認する。

## 2. PRD境界を確定する

必ず `$grill-me` の一問一答で、PRDを「独立して成功判定できる1つのユーザー成果」へ絞る。次の規則を適用する。

- 同じ原因でも、期待成果または成功判定が独立するなら別PRDにする。
- 複数の症状、画面、操作、対象ユーザーがあっても、1つの成果と成功判定へ収束するなら同じPRDにする。
- 対象ユーザー、技術領域、実装量、期間の違いだけでは分割しない。
- UI、API、DB、外部連携などの技術的分割は、後続の設計とTasksへ委ねる。
- 複数のSuccess Metricsが同じ成果を異なる側面から測るなら分割しない。
- 既存機能改善と新規成果は、独立して成功判定できる場合だけ分割する。
- 別PRDへの依存があっても統合しない。依存先だけではユーザー価値が成立しない作業は、同一PRDのTechnical Story候補として扱う。
- 同じ成果に必須でない追加価値は含めず、別PRD候補またはFollow-upにする。
- 規制、安全性、セキュリティ、データ整合性など成果成立に不可欠な制約は同じPRDに含める。

複数の独立した成果が見つかった場合は、候補ごとに次を提示する。

- ユーザー成果
- 境界
- 成功判定
- 依存関係
- 推奨する今回の対象と理由

人間に今回作成するPRDを1つだけ確定してもらう。選ばれなかった候補を現在のPRDへ混ぜない。

## 3. PRD内容を確定する

少なくとも次を人間と確定する。

- Goal
- Problem
- Target User
- Scope
- Out of Scope
- User Value
- Success Metricsの評価方法と観測時期

`Scope` には、成果成立のために変更または調査が必要な対象領域までを書く。画面、API、データ、運用などは対象領域として記載できるが、具体的なUI、エンドポイント、スキーマ、ライブラリなどの解決策は書かない。境界が曖昧な項目を推測で含めない。

`Out of Scope` には、誤って今回の対象と解釈される可能性がある項目だけを書く。無関係な項目を網羅しない。

AIが事実や方針を補完してはならない。未確定事項は次のように分類する。

- 人間の判断が必要: `Open Questions`
- 人間が暫定的に許容した前提: `Assumptions`
- 資料またはコードから確認できる事実: 調査して該当項目へ反映
- AIの推測: 本文へ記載しない

PRD境界、Goal、Problem、Target User、Scopeの未確定事項はBlockingとして扱い、解決するまで生成しない。補助的な未確定事項は、PRDの後続利用を妨げない場合だけ `Open Questions` に残してよい。

## 4. IDと保存先を確定する

既存の `docs/prds/` を調査し、次の形式で候補を提示する。

- PRD ID: `prd-XXX-<slug>`
- 保存先: `docs/prds/<prd-id>/prd.md`

連番は既存PRDと重複しない次の番号を候補にする。slugは解決策名ではなく、ユーザー課題または期待成果を表す短い英語にする。重複、欠番、命名の衝突を黙って解消せず、候補と理由を示して人間の確定を得る。

既存PRDの更新ではIDと保存先を維持する。分割が必要になった場合は元PRDを自動分割または統合せず、元PRDへ残す成果、新PRD候補、Scope・Success Metrics・依存関係・後続成果物への影響を提示する。元PRDの更新と新PRDの作成は別の実行として扱う。

## 5. 生成承認を得る

Draft生成前に次を確認する。

- 1つのユーザー成果とPRD境界が確定している
- Goal、Problem、Target User、Scopeが確定している
- Success Metricsに評価方法と観測時期がある
- 後続利用を妨げるOpen Questionsが残っていない
- PRD IDと保存先を人間が確定している

確定事項、Assumptions、残るOpen Questions、選ばれなかったPRD候補を要約し、「議論を終了してDraftを生成するか」を確認する。人間が明示的に承認するまでファイルを書き換えない。

議論履歴を別ファイルへ保存しない。重要判断の履歴が必要な場合だけ、生成後にADR候補として提示する。

## 6. Draftを生成する

テンプレートの見出し構成と順序を維持する。該当しない項目も削除せず `Not applicable` と記載する。テーマ固有の補足は既存項目内へ記述し、独自見出しは人間が承認した場合だけ追加する。

Statusは常に `Draft` とする。既存のStatusが `Accepted` または `Implemented` でも `Draft` へ戻し、`Status Reason` に変更テーマと再レビューが必要な理由を書く。`Accepted` への変更は行わない。

更新時は今回のテーマに関係する箇所だけを変更する。整理目的の削除、言い換え、再構成をしない。後続の `backlog.md`、`requirements.md`、ADR、`architecture-change.md`、`tasks.md` は自動更新しない。

本文へ出典欄を追加しない。根拠は調査した資料と今回の議論で人間が確定した内容に限定する。

## 7. 自己検証する

生成後に次だけを確認し、必要なら修正する。

- テンプレートの見出しと順序を維持している
- Statusが `Draft` で、`Status Reason` が適切である
- 1つの独立して成功判定できるユーザー成果に限定されている
- 解決策が混入していない
- ScopeとOut of Scopeの境界が確定内容と一致している
- Open QuestionsとAssumptionsを正しく分類している
- 入力資料または人間が確定した内容と矛盾していない
- 更新時にテーマ外の既存内容を意図せず変更していない
- PRD IDと保存先が一致している

内容の妥当性レビューとAccepted化は行わず、別セッションの `document-review` へ委ねる。

## 8. 完了を報告する

次を簡潔に報告する。

- 作成または更新した `prd.md` のパス
- PRD ID、対象ユーザー成果、Status
- 参照した資料
- 残るOpen QuestionsとAssumptions
- 選ばれなかったPRD候補
- 影響を受ける既存成果物と矛盾
- ADR候補
- 次工程として別セッションで `document-review` が必要であること

選ばれなかった候補は自動保存せず、独立した別PRD候補として完了報告へ提示する。
