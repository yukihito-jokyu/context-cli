# 仕様書定義 (Spec Definition)

ユーザー課題を独立して検証可能な成果に絞り、要件と初期技術設計（処理フロー、パッケージ境界など）を統合した `spec.md` を作成する。本スキルでは、実装開始前のすべての意思決定をこの1ファイルにまとめる。

## 入力

- 新規作成: 解決したい課題・要件
- 更新: 対象仕様書の3桁IDと変更方針

## 1. 現状を調査する

`docs/product.md`、`docs/engineering/*.md`、関連するコード、既存の仕様書を調査する。コードや既存設定から判明する事実はユーザーに質問せず、自分で調べる。

## 2. grill-me スキルを用いた一問一答の議論

[grill-me](../grill-me/SKILL.md) に従い、最終的な `spec.md` に必要な情報を引き出す。質問項目を固定せず、対象機能に応じた意思決定を掘り下げる。

（※仕様書に含めるべき主要要素：）

1. 対象ユーザーと解決したい課題
2. ゴールと成功指標
3. スコープとスコープ外
4. インターフェース・入出力仕様（CLI引数/フラグ、stdin、stdout/stderr、終了コード、ファイル変更などの具体的な入出力）
5. 主要処理フロー（正常系・異常系のロジック分岐）
6. [AGENTS.md](../../../AGENTS.md) に適合するファイル配置と責務
7. 制約事項、非機能要件、前提条件

## 3. リスクの検討

データ、既存機能、セキュリティへの影響を評価し、レビュー時の注意点を記録する。

## 4. 仕様書ファイルを組み立てる

対話で確定した例外系、設計判断、懸念、背景を失わず、適切なセクションへ記録する。

テンプレート [spec.md](../../../docs/ai-driven-development/templates/spec.md) をベースにして、`docs/specs/spec-XXX-<slug>/spec.md` を次の構成で作成する。

- 仕様書ID（例: `spec-001`）
- ゴール、課題、対象ユーザー、ユーザー価値
- 成功指標
- スコープ、スコープ外
- ユーザーストーリー
- 完成条件
- インターフェース・入出力仕様（CLI引数、stdin、stdout、stderr、終了コード、ファイル変更など）
- 制約事項
- 非機能要件
- リスク (本実装が及ぼすリスク影響の1文メモ)
- 前提条件
- 未解決事項
- **技術設計ドラフト（全体概要）**
  - **処理フローチャート（概要）**: システム全体の主要な処理の流れ（詳細な条件分岐は各タスク側に記載。Mermaid記法推奨）
  - **シーケンス図（概要）**: 主要なモジュール・コンポーネント間の相互作用の全体像（Mermaid記法推奨）
  - **ファイル配置方針**: 新規作成・変更が想定される主なパッケージや大枠のファイル配置方針とそれぞれの責務

## 5. 生成する

生成前に主要要件、技術設計の骨子、リスクを要約して人間に提示し、生成の承認を得る。承認後、ファイルを生成する。
完了時は、出力先、リスク、および次フェーズの `review-spec` を報告する。
