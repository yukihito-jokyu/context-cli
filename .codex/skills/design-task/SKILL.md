# タスク個別設計

特定の「処理フロー（タスク）」を開始する前に、その仕様とアーキテクチャ設計を人間と議論して確定させ、詳細なタスク設計書を完成させる。

## 入力

- 設計対象のタスクファイルパス（例: `docs/specs/spec-XXX-<slug>/tasks/01-init.md`）

## 1. 関連文脈を読み込む

対象タスクファイル、`spec.md`、および `docs/engineering/*.md`、さらに既存のコード（特に依存・接続する予定のパッケージやインターフェース）を調査する。

## 2. grill-me で一問一答の壁打ちを行う

[grill-me](../grill-me/SKILL.md) に従い、次の設計判断を確定する。

1. フローのトリガー、入力、出力。
2. 正常系、異常系、ロールバック。
3. [AGENTS.md](../../../AGENTS.md) に適合するファイル配置と責務。
4. E2E、統合、単体テストの対象と実行方法。

## 3. タスクファイルを詳細化する

壁打ちの合意結果に基づいて、対象のタスクファイルを以下の見出し構成で更新する。

### 更新テンプレート:

```markdown
# タスク名 (例: 01-init-command)

- **ステータス**: 未着手 (Pending)
- **対象ストーリー**: (仕様書の Story ID)

## 1. 処理フローチャート (Flowchart)

<!-- Mermaidでビジネスロジックの条件分岐や正常・異常系を表すフローチャートを記述 -->

## 2. シーケンス図 (Sequence Diagram)

<!-- Mermaidで pkg/cmd や internal 配下の内部モジュールをまたぐオブジェクト間の相互作用を表現 -->

## 3. ファイル配置・責務定義

<!-- どのディレクトリのどのファイルに何を配置するか、変更・新規作成の予定を記載 -->

- `[NEW/MODIFY]` [ファイル名](リンク) : このファイルの役割と実装する処理

## 4. 実装チェックリスト

- [ ] テストコードの作成 (TDD)
- [ ] コマンドおよび内部モジュールの実装
- [ ] 結合テストおよび動作確認

## 5. テスト・検証計画

- **E2E/結合テスト方法**: (具体的なコマンドやシナリオ)
- **単体テスト対象**: (ロジックが複雑な部分)
```

**※注意**: ファイルヘッダーのステータスは `未着手 (Pending)` のままとする。

## 4. 完了報告

- 生成・詳細化した設計内容の要約を人間に報告し、次フェーズの `review-task-design` に進むことを提案する。
