# タスク個別設計

特定の「処理フロー（タスク）」を開始する前に、その仕様とアーキテクチャ設計を人間と議論して確定させ、詳細なタスク設計書を完成させる。

## 入力

- 設計対象のタスクファイルパス（例: `docs/prds/prd-XXX-<slug>/tasks/01-init.md`）

## 1. 関連文脈を読み込む

対象タスクファイル、`prd.md`、`docs/decisions/adr-summary.md`、`docs/engineering/*.md`、および既存のコード（特に依存・接続する予定のパッケージやインターフェース）を調査する。

## 2. grill-me で一問一答の壁打ちを行う

以下の順序に従い、**一度に1つずつ質問する**ことで、人間からタスクの要件を吸い出す。各質問には推奨される設計案とその理由を必ず提示する。

1. **フローのトリガーと入出力**: 何を契機に起動し、どんなパラメーターを受け取り、ユーザーにどう応答するか。
2. **正常系・異常系のロジック**: 途中でエラーが発生する条件と、その際のエラー処理・ロールバック要件。
3. **Goパッケージの配置と責務**: `adr-summary.md` のレイヤー境界に従い、どのパッケージ（`cli`, `application`, `domain`, `infrastructure`）に何の構造体/関数/インターフェースを配置するか。
4. **テスト方針**: どのようなE2E/統合テストでフロー全体を検証し、どのロジックを単体テストにするか。

## 3. タスクファイルを詳細化する

壁打ちの合意結果に基づいて、対象のタスクファイルを以下の見出し構成で更新する。

### 更新テンプレート:

```markdown
# タスク名 (例: 01-init-command)

- **ステータス**: 未着手 (Pending)
- **対象ストーリー**: (PRDの Story ID)

## 1. 処理フローチャート (Flowchart)

<!-- Mermaidでビジネスロジックの条件分岐や正常・異常系を表すフローチャートを記述 -->

## 2. シーケンス図 (Sequence Diagram)

<!-- Mermaidで Go の各パッケージ境界（Clean Architecture）をまたぐオブジェクト間の相互作用を表現 -->

## 3. ファイル配置・責務定義

<!-- どのディレクトリのどのファイルに何を配置するか、変更・新規作成の予定を記載 -->

- `[NEW/MODIFY]` [ファイル名](リンク) : このファイルの役割と実装する処理

## 4. 実装チェックリスト

- [ ] テストコードの作成 (TDD)
- [ ] 各レイヤーの実装
- [ ] 結合テストおよび動作確認

## 5. テスト・検証計画

- **E2E/結合テスト方法**: (具体的なコマンドやシナリオ)
- **単体テスト対象**: (ロジックが複雑な部分)
```

**※注意**: ファイルヘッダーのステータスは `未着手 (Pending)` のままとする。

## 4. 完了報告

- 生成・詳細化した設計内容の要約を人間に報告し、次フェーズの `review-task-design` に進むことを提案する。
