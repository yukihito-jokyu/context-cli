# タスク一括設計 (Design Tasks)

仕様書の要件と技術設計ドラフトを元に、全体を無駄なく重複のないタスクに分解し、それぞれに対して最初から具体的な設計を記述したタスクファイルを一括生成する。

## 入力

- 対象の仕様書ID（および `spec.md` のパス）

## 1. 関連文脈を読み込む

仕様書（`spec.md`）、`docs/engineering/*.md`、および既存のコードベース（特に再利用・拡張する予定のパッケージやインターフェース）を調査する。

## 2. grill-me スキルを用いた一問一答の議論

[grill-me](../grill-me/SKILL.md) に従い、以下の設計判断とタスク分解の構成を確定する。

1. 各タスクにおける具体的な処理フロー、正常系・異常系・境界条件。
2. [AGENTS.md](../../../AGENTS.md) に適合する、各タスクにおける具体的なファイル配置と責務。
3. テスト（E2E、統合、単体テスト）の検証計画。
4. **タスク分解の構成（無駄や重複のない、縦スライスされた機能フローごとの切り出し）**

## 3. タスク分解と重複排除の原則

タスクを分解する際は、以下の原則を厳格に適用する。

- **縦スライス（実装＋テストの一体化）**:
  - 要件を、レイヤーで横割りにする（例：「APIを実装するタスク」「テストを書くタスク」のように分ける）のではなく、**「ユーザー操作から応答までの縦スライスの処理の流れ」**ごとに分割する。
  - **一つの流れの実装とテストコードの作成は、必ず同一のタスクに含める。**
- **無駄・重複の排除**:
  - 共通で利用するヘルパー、ユーティリティ、共通のバリデーションロジック等は、独立したタスクに分けず、**最初にそれを使用するタスクの中に含める**か、あるいは**共通の基盤モジュールを作成する単一の依存タスクとして定義**する。
  - 後続のタスクで同じ実装やテストを繰り返さないよう、各タスクの責務を明確に切り分ける。
- **機能的な独立性**:
  - 機能的に複数ある場合は適切にタスクを分ける。1つのタスクがカバーするビジネスロジックが肥大化しすぎないようにする。

## 4. 具体設計を含んだタスクファイルを生成する

`docs/specs/spec-XXX-<slug>/tasks/` ディレクトリを作成し、その中に `01-<name>.md`、`02-<name>.md` ... と依存順に連番のファイルを生成する。
各ファイルは、以下のテンプレートに従い、**最初から「具体設計（詳細仕様・実装チェックリスト・テスト計画）」を記述した状態**で生成する。

### 生成テンプレート (01-xxx.md):

```markdown
# タスク名 (例: 01-init-command)

- **ステータス**: 未着手 (Pending)
- **対象ストーリー**: (仕様書の Story ID, 例: ST-001)

## 1. 処理フローチャート (Flowchart)

<!-- Mermaidでこのタスクが対象とする具体的なビジネスロジックの条件分岐や例外処理のフローを記述。単純なタスクでspec.mdの全体フローと同一になる場合は、図を省略し「spec.mdを参照」としてもよい -->

## 2. シーケンス図 (Sequence Diagram)

<!-- Mermaidでこのタスクで実装・修正する範囲の、モジュール・クラス・関数間の具体的な相互作用を表現。不要な場合は、図を省略し「spec.mdを参照」としてもよい -->

## 3. ファイル配置・責務定義

<!-- このタスクで実際に新規作成または修正する具体的なファイルパスと、それぞれの詳細な実装内容・役割を記載 -->

- `[NEW/MODIFY]` [ファイル名](リンク) : このファイルの役割と実装する処理

## 4. 実装チェックリスト

- [ ] テストコードの作成 (TDD)
- [ ] 内部モジュールの実装
- [ ] 結合テストおよび動作確認

## 5. テスト・検証計画

- **E2E/結合テスト方法**: (具体的なコマンドやシナリオ)
- **単体テスト対象**: (ロジックが複雑な部分)
```

## 5. 完了報告

- 切り出したタスクの一覧（連番とタイトル、対象ストーリー）とそれぞれの設計要約を人間に報告する。
- 全タスクの設計が完了したため、次のフェーズ `review-tasks-design` に進むことを提案する。
