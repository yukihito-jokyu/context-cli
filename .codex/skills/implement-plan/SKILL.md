# 計画実装

承認済み計画を、不要な停止と文書更新を挟まずにコードへ反映する。

## 必須条件

- PRDの3桁IDを受け取る。
- `prd.md`と`implementation-plan.md`が`Accepted`である。
- 必要なADRとEngineering Foundationが`Accepted`である。
- Blocking Open Questionがない。

## 1. 正本と差分を読む

PRD、計画、Engineering Foundation、関連ADR、現在Architecture、関連コード・テスト・設定、Git状態を読む。既存の未コミット変更を識別して保持する。

開始時に差分基準、対象Work Unit、品質ゲート、リスクを簡潔に報告する。通常の実装開始に追加承認を求めない。

## 2. 連続実装する

Work Unitの依存順に、PRD全体を通して次を繰り返す。

1. 対応ACとテスト方針を確認する
2. 必要なテストを先に追加し、可能なら失敗を確認する
3. Scope内で最小限に実装する
4. 対象テストを成功させる
5. 影響範囲の回帰テストを行う
6. 差分を自己レビューする
7. 次の実行可能なWork Unitへ進む

Work Unitごとの人間承認、Status変更、コンテキスト圧縮、`implementation-plan.md`への実行ログ追記は行わない。実行結果の正本はコード、テスト、Git差分、CIとする。

## 3. 停止条件

次の場合だけ停止する。

- Scope、AC、制約の変更が必要
- ADRまたは重要設計判断の変更が必要
- 破壊的操作、新規外部依存、本番・共有環境操作が必要
- 既存差分と安全に分離できない
- 重大な検証ができず、代替確認もできない
- 同じ原因で3回失敗し、Scope内の別手段がない

停止時は原因、影響、試行、戻るフェーズ、解除条件を報告する。局所的な実装詳細やファイル候補の差異は、計画の意図を変えない限り自律的に調整する。

## 4. 全体検証する

全Work Unit後に、対象テスト、全体テスト、Lint、型検査、ビルド、セキュリティ検査など`Quality Gates`を実行する。実行不能項目は理由、代替検証、残存リスクを記録する。

PRD、計画、ADR、Engineering Foundationへの適合、Scope外変更、エラー処理、セキュリティ、データ整合性、テスト不足を自己レビューする。

## 5. 完了候補を報告する

文書へ詳細ログを残さず、完了報告に次を含める。

- 変更ファイルと主要実装
- ACごとの検証概要
- 実行した品質ゲート
- 未実行項目と残存リスク
- 計画との差異
- 既存差分との区別

コミット、PR、push、デプロイは指示がない限り行わない。次フェーズは`review-implementation`とする。
