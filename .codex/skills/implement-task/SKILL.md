# タスク実装

設計合意されたタスク（フロー）の要件を、不要な一時停止を挟まずに確実かつ安全にコードへ反映する。

## 入力

- 実装対象のタスクファイルパス（例: `docs/prds/prd-XXX-<slug>/tasks/01-init.md`）

## 1. 準備

- 対象タスクファイル、`prd.md`、`docs/decisions/adr-summary.md`、`docs/engineering/*.md`、および関連する既存コード・テストファイルを読み込む。
- 開始時に、対象のタスクファイルヘッダーのステータスを `- **ステータス**: 仕掛中 (In Progress)` に変更する。

## 2. サイクル実行

タスクのチェックリストと設計計画に従い、以下のサイクルを繰り返す。

1. **テストの先行追加 (Test First)**: AC（完成条件）や技術的制約に対応するテストを追加し、可能であれば未実装を理由として失敗（Red）することを確認する。
2. **最小限の実装**: テストを通すために必要な最小限のコードを記述する。
3. **テストパスの確認 (Green)**: 実装したテスト、および既存のテストスイートを実行し、すべて正常にパスすることを確認する。
4. **リファクタリング**: 変更した範囲内の設計を整理・リファクタリングする。
5. **回帰確認**: `task test` や `task lint` などの品質ゲートを実行し、他への影響がないことを確認する。

## 3. 進捗更新

- 作業の進捗に合わせて、タスクファイル内の `## 4. 実装チェックリスト` のチェックマーク `[ ]` を `[x]` に随時更新する。
- 実装の過程で判明した変更ファイルやテスト結果を `## 3. ファイル配置・責務定義` や `## 5. テスト・検証計画` に反映する。

## 4. 完了

- すべての実装・検証が完了し、`task lint` および `task test` がパスしたら、タスクファイルヘッダーのステータスを `- **ステータス**: 完了 (Completed)` に更新する。
- 変更ファイルの一覧と、テスト検証結果の証跡を添えて人間に完了報告を行う。
- 次フェーズの `review-implementation` を提案する。
