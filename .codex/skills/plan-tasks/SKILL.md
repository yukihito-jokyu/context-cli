# タスク分割

仕様書の要件とユースケースを、独立して実装・検証可能な「1つの処理フロー」ごとに分割し、開発の全体スケジュール（ロードマップ）を作る。

## 入力

- 対象の仕様書ID（および `spec.md` のパス）

## 1. 依存関係とフローを特定する

- `spec.md` のユーザーストーリー、制約事項、および技術設計、`docs/engineering/development-rules.md` などの制約を読み込む。
- 要件を、レイヤーで横割りにするのではなく、**「ユーザー操作から応答までの縦スライスの処理の流れ」**ごとに分割する。
- 各フローを実装するための依存順序（前提となるタスクがどれか）を決定する。

## 2. スケルトンファイルを生成する

`docs/specs/spec-XXX-<slug>/tasks/` ディレクトリを作成し、その中に `01-<name>.md`、`02-<name>.md` ... と依存順に連番のファイルを生成する。各ファイルの初期記述は、以下のテンプレートに従う最小限のスケルトンとする。

### 生成テンプレート (01-xxx.md):

```markdown
# タスク名 (例: 01-init-command)

- **ステータス**: 未着手 (Pending)
- **対象ストーリー**: (仕様書の Story ID, 例: ST-001)

※このタスクの設計は、タスク開始時に `design-task` スキルを実行して詳細化する。
```

**※注意**: この時点では、フローチャート、シーケンス図、具体的なファイル配置計画などの「詳細設計」は一切書き込まない（コンテキスト爆発の防止）。

## 3. 完了報告

- 切り出したタスクの一覧（連番とタイトル、対象ストーリー）を人間に報告する。
- 次に開始すべき最初のタスク（例: `01-<name>.md`）を指定し、`design-task` に進むことを提案する。
