# 実装コードレビュー

実装された機能コードおよびテストコードの品質を担保し、不具合の流出を防止する。

## 入力

- 対象のタスクファイルパス（例: `docs/prds/prd-XXX-<slug>/tasks/01-init.md`）
- 変更されたソースコードおよびテストコードのGit差分

## 1. レビュー観点

以下の観点で実装内容を厳密にレビューする。

1. **要件と設計の適合性**:
   - `prd.md` の対象AC（完成条件）が漏れなく満たされていることを確認する。
   - `01-<name>.md` で合意された処理フローやファイル配置計画に沿って正しく配置・実装されていることを確認する。
2. **依存方向とモジュール境界 (structure.md)**:
   - `pkg/cmd/` と `internal/` の依存方向が正しく保たれていることを確認する。`internal/` 配下のパッケージが `pkg/cmd/` に依存していないこと。
   - 各コマンドの依存関係（I/Oや設定等）が、`Factory`（`pkg/cmd/factory.go`）を経由して注入されていることを確認する。
3. **コード品質と堅牢性**:
   - エラーハンドリングが適切であり、エラーが途中で捨てられておらず、かつ呼び出し元で識別可能であることを確認する。
   - リソースリーク（ファイルオープン、ロック解除等）がないことを確認する。
   - すべてのGoソースコード内のコメントが日本語で記述されていることを確認する。
   - Goの一般的な命名規約や構造に従っていることを確認する。
4. **テストの品質**:
   - 正常系・異常系・境界条件がテストコードでカバーされていることを確認する。
   - 各コマンドの単体テストが、Cobraを経由せずに `Options.Run()` を直接呼び出すテーブル駆動テスト形式で記述されていることを確認する。
   - テストにおいて、テスト用一時ディレクトリ（`t.TempDir()` など）を使用し、利用者の実設定を変更していないことを確認する。

## 2. フィードバックと改善

- 指摘事項（バグ、不整合、改善案）を明確に一覧化し、人間に提示する。
- 改善案に人間が合意した場合、`implement-task` の要領で修正し、再度テストを実行する。
- 修正が完了し、指摘がすべて解消されたら、レビュー完了を報告する。
- 全タスクの実装・レビューが完了している場合は、`update-architecture` に進むことを提案する。
