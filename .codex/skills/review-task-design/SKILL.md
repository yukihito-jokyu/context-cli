# タスク設計レビュー

個別タスクの設計（フローチャート、シーケンス図、配置計画）を検証し、`implement-task` へ安全に進められる品質であることを担保する。

## 入力

- レビュー対象のタスクファイルパス（例: `docs/prds/prd-XXX-<slug>/tasks/01-init.md`）

## 1. レビュー観点

以下の技術制約および設計品質を満たしているかを厳格に検証する。

1. **依存方向とモジュール境界 (structure.md)**:
   - `pkg/cmd/` および `internal/` 配下の各パッケージが正しい依存方向を守っていることを確認する。`internal/` 配下のパッケージは `pkg/cmd/` に依存してはならない。
   - `internal/` 配下の各パッケージ間で循環依存が発生していないことを確認する。
   - 依存関係（入出力ストリーム、設定のローダー等）は、すべて `Factory`（`pkg/cmd/factory.go`）を経由して引き渡されていることを確認する。
2. **責務配置の妥当性**:
   - Cobraコマンド、フラグ定義、UIモデルが `pkg/cmd/` に配置されていることを確認する。
   - 外部非公開の独自ユーティリティやヘルパーモジュールが、`internal/` 配下の単一責任のパッケージ（例: `internal/config` など）として独立して配置されていることを確認する。
   - `utils`、`common`、`helpers` のような曖昧な名前のパッケージが作成されていないことを確認する。
3. **ロジックの網羅性**:
   - フローチャートおよびシーケンス図において、正常系だけでなく例外（エラー）発生時の処理・呼び出し元の制御をカバーしていることを確認する。
4. **テスト設計**:
   - テスト計画に、E2E/結合テストで動作確認するための具体的な実行手順・コマンドが明記されていることを確認する。
   - 単体テストの範囲が明確であることを確認する。

## 2. フィードバックと改善

- 指摘（懸念点・ルール違反）がある場合は、改善策とともに人間に提示する。
- 改善案に人間が合意した場合、対象のタスクファイルを直接修正する。
- 設計が問題ないレベルに到達したら、人間がステータスを `仕掛中 (In Progress)` または `承認 (Accepted)` にして実装に進むことを推奨する。
