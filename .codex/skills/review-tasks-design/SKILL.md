# 全タスク設計レビュー (Review Tasks Design)

生成されたすべてのタスク設計書を第三者視点で検証し、実装フェーズへ進める品質かを判定する。

## 入力

- レビュー対象のタスクファイル群が格納されたディレクトリパス（例: `docs/specs/spec-XXX-<slug>/tasks/`）

## 1. レビュー観点

次を確認する。

- **ストーリーの網羅性**:
  - `spec.md` に定義されたすべてのユーザーストーリーおよび完成条件が、漏れなくタスクに割り振られているか。
- **縦スライス構成 (実装＋テストの一体化)**:
  - 各タスクが「実装フェーズ」「テストフェーズ」のように横割りにされておらず、1つの機能フローの実装とテストコードの作成が同一タスク内で完結するよう計画されているか。
- **重複・無駄の排除**:
  - 各タスク間で同一の実装やテストの重複、あるいは無駄なタスク切り出しがないか。
  - 共通で利用するヘルパーやユーティリティ等は、最初に使用するタスク、もしくは共通基盤タスクに適切に集約されているか。
- **適切な機能分割**:
  - 機能的に複数ある場合は適切にタスクが分かれているか。1つのタスクが大きくなりすぎていないか。
- **規約適合性**:
  - 計画されているファイル配置と責務定義が [AGENTS.md](../../../AGENTS.md) のアーキテクチャやルールに適合しているか。
- **テスト計画の具体性**:
  - テスト計画に具体的なシナリオ、テスト対象、およびテスト実行コマンドが記述されているか。

## 2. フィードバックと改善

- 重大度、根拠、改善案を整理して親エージェントへ報告する。
- サブエージェントはファイルやステータスを直接変更しない。
- レビューがすべて合格し、ユーザーから設計承認を得た後、実装フェーズへ進むことを明示する。
