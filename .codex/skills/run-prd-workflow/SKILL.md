# PRD標準フロー実行

新規PRDの標準フローを進行管理し、各工程の作業を担当する子スキルへ委譲する。

## 必須条件

- 新規PRDで解決したい1つのユーザー課題を必須入力とする。再開時はPRDの3桁連番IDも受け取る。
- 新規PRD専用とし、既存PRDの更新には使用しない。
- 標準フロー専用とする。開始時のリスク評価で軽量フローまたは緊急変更フローが適切なら、理由と使用すべき個別フローを提示して停止する。
- このスキルは進行管理だけを担当する。成果物作成、レビュー、実装、修正、architecture更新、release-checkは対応する子スキルへ委譲する。
- 子スキルの開始条件、入力、停止条件、人間承認、変更可能範囲を緩和しない。
- 会話要約を正本にしない。再開時は永続成果物、Git差分、テスト結果を読み直す。
- 実際のリリース、デプロイ、本番操作、リリース後検証は行わない。

## 1. 開始または再開を判定する

PRD IDがなく、対象PRD成果物も存在しない場合は新規開始とする。ユーザー課題が曖昧なら、最初の `$grill-me` で1つの独立して成功判定できるユーザー成果へ絞る。

<!-- 固定の定義または対応関係を1行で維持するため。 -->
<!-- textlint-disable preset-ja-technical-writing/sentence-length -->

PRD IDがある場合は、対象ディレクトリと成果物のStatus、Open Questions、レビュー結果、`tasks.md` のTask状態、Architecture Update、Release Checkを読み、未完了の最初の工程を再開候補にする。

<!-- textlint-enable preset-ja-technical-writing/sentence-length -->

複数の再開候補、矛盾、未記録の変更がある場合は自動選択しない。候補、根拠、影響を提示し、人間が再開地点を確定する。

既存PRDのGoal、Problem、Scopeを変更する必要がある場合は、この統合フローを停止して個別スキルへ委譲する。

## 2. 適用フローを確定する

ユーザー影響、データ変更、認証・認可、課金、外部連携、影響範囲、仕様の明確さ、障害時の運用影響を評価する。

標準フローを推奨する根拠を提示し、人間の確定後だけ開始する。軽量または緊急変更フローを推奨する場合は、このスキルでは進めない。

作業中に標準フローの前提を損なう変更が判明した場合も、理由と移行先を提示して停止する。

## 3. 工程表に従う

次の順序で進める。各矢印は開始条件を満たす場合だけ進む。

1. `$grill-me` でPRD境界と内容を議論する
2. 同じコンテキストで `$create-prd` を実行し、`prd.md` Draftを生成する
3. 圧縮後、`$document-review` で `prd.md` を重レビューし、人間がAcceptedを確定する
4. `$create-backlog` で `backlog.md` Draftを生成する
5. 圧縮後、`$document-review` で軽レビューし、人間がAcceptedを確定する
6. `$create-requirements` で `requirements.md` Draftを生成する
7. 圧縮後、`$document-review` で重レビューし、人間がAcceptedを確定する
8. `$extract-adr-candidates` を必ず実行して `adr-candidates.md` Draftを生成する
9. 圧縮後、`$document-review` で最重レビューし、候補ごとのStatusを人間が確定する
10. Accepted候補がある場合だけ、`$create-adr` で各ADR Draftを生成する
11. ADRを生成した場合は圧縮後、各ADRを `$document-review` で最重レビューし、人間がAcceptedを確定する
12. `$create-architecture-change` で `architecture-change.md` Draftを生成する
13. 圧縮後、`$document-review` で重レビューし、人間がAcceptedを確定する
14. `$create-tasks` で `tasks.md` Draftを生成する
15. 圧縮後、`$document-review` で軽レビューし、人間がAcceptedを確定する
16. `$implementation` で実行可能なTaskを一件だけ実装し、Completion Candidateを提示する
17. 人間がTaskのCompletedを確定し、`tasks.md` へ記録した後に圧縮する
18. 未完了Taskがあれば手順16へ戻る
19. 全Task完了後に圧縮し、`$code-review` でPRD全体をレビューする
20. 人間がコードレビュー完了を確定した後、`architecture-change.md` のImplemented候補を確認し、人間が確定する
21. `$update-architecture` を実行する
22. 圧縮後、変更した `docs/architecture/*.md` を `$document-review` で中レビューし、人間がAcceptedを確定する
23. `$release-check` を実行し、人間がリリース可否を確定する
24. 承認後、リリース後検証項目とFollow-up手順を提示して終了する

初回PRDで `docs/architecture/*.md` が存在しないことは許容する。手順21で標準6ファイルを作成する。

## 4. 永続成果物の境界で圧縮する

圧縮は子スキルの呼び出し回数ではなく、状態を永続成果物へ保存できた工程境界で要求する。

圧縮前に次を確認する。

- 子スキルの工程が完了または明示的にBlockedになっている
- 必要な人間判断とStatus確定が完了している
- 成果物、`tasks.md`、テスト結果などの正本へ状態が記録されている
- 次工程、必須入力、再開条件が一意である

確認後、次工程名、再開に必要なPRD IDまたは対象パスを提示し、ユーザーへコンテキスト圧縮を依頼して停止する。圧縮前に次工程を開始しない。

`$grill-me` と `$create-prd` は同一セッションの議論内容を必要とするため、`prd.md` Draft生成までは圧縮しない。

## 5. 圧縮後に独立レビュー相当の状態を作る

この統合スキル経由に限り、コンテキスト圧縮後の同一セッションで `$document-review` と `$code-review` を実行できる。

レビュー開始前に必ず次を行う。

- 対象成果物または実装差分を正本から再読込する
- 上流成果物、関連ADR、Engineering Foundation、レビュー観点、テスト結果を再読込する
- 作成・実装時の推論、自己評価、会話要約をレビュー根拠にしない
- 圧縮後であることと、再読込した資料を開始概要に明記する

条件を満たせない場合は同一セッションでレビューせず、新しいセッションを要求する。圧縮せずに作成・実装からレビューへ続行してはならない。

## 6. 人間のゲートを守る

子スキルが要求する生成承認、レビュー開始、Status変更、Task完了、コードレビュー完了、Implemented確定、リリース可否は人間が確定する。

統合スキルは承認を推測せず、未確定のゲートを越えない。人間の回答待ちである場合は次工程を開始しない。

## 7. 条件分岐を処理する

- ADR候補抽出は常に実行する。
- `create-adr` とADRレビューは、Accepted候補がある場合だけ実行する。
- `docs/architecture/*.md` の確認は既存ファイルがある場合だけ設計入力に含める。
- 実装はTaskごとに区切り、各TaskのCompleted確定後に圧縮する。
- product更新、Engineering Foundation作成・更新はこのフローに含めない。必要性が判明した場合は統合フローを止め、個別スキルへの移行候補を提示する。

工程省略が必要な場合は、対象、理由、影響、残存リスク、代替確認を提示し、人間の明示承認を得る。子スキルが省略を禁止する工程は省略しない。

## 8. Blockedと差し戻しを処理する

子スキルが上流成果物への差し戻しを要求した場合、自動修正しない。

1. 差し戻し理由、影響範囲、現在工程を提示する
2. 戻る成果物、担当する子スキル、必要な再レビュー、下流への影響を提示する
3. 人間が差し戻し先を確定する
4. 担当成果物または `tasks.md` の許可された欄へ状態を記録する
5. コンテキスト圧縮を依頼して停止する
6. 圧縮後、指定された子スキルから再開する

修正後は、影響を受ける下流成果物を順に再レビューする。問題が解消したと推測して元の工程へ飛ばない。

## 9. 完了する

`release-check` が `Pass` となり、人間がリリース可否を承認した後、次を提示する。

- リリース後に確認する主要ACとSuccess Metrics
- 監視、ログ、エラー率、移行結果などの確認項目
- 切り戻し・復旧条件と手順
- Follow-upの記録先と必要項目
- 実際のリリースとリリース後検証は責務外であること

ここで統合スキルを終了する。リリース後検証結果やFollow-upは、必要に応じて個別に正本へ記録する。
