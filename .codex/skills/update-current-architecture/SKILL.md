# 現在Architecture更新

実装済みの現在状態を、後続開発が参照できる最小限のArchitecture文書へ反映する。

## 必須条件

- PRDの3桁IDを受け取る。
- 実装レビューでCriticalとHighが解消済みである。
- PRD、実装計画、必要なADRがAcceptedである。

## 1. 更新範囲を決める

実装差分、コード、設定、テスト、ADR、Engineering Foundation、既存`docs/architecture/*.md`を読む。

- 初回はシステム全体を説明するために必要な最小ファイルだけを作る。
- 既存文書がある場合は今回の差分で影響を受けるファイルだけを更新する。
- 空の標準ファイルや`Not applicable`専用ファイルを数合わせで作らない。
- 未実装の予定、変更履歴、判断理由は記載しない。判断理由はADRを参照する。

## 2. 現在状態を記述する

必要に応じて次を既存構成へ反映する。

- システム概要と主要フロー
- API・CLI境界
- ドメインと責務
- パッケージと依存方向
- データ・永続化
- 外部連携、設定、運用、復旧
- セキュリティ境界

ファイル分割は情報量と変更頻度に基づいて決める。既存テンプレートがある場合は利用するが、該当しない文書の作成を強制しない。

## 3. 不一致を処理する

コードがPRD、計画、ADR、Engineering Foundationに反する場合は文書で正当化せず停止する。単なる記載漏れや実装名の確定だけを反映する。

## 4. 自己検証する

- コードと設定の現在状態に一致する
- 文書間に矛盾がない
- 変更ファイルだけを更新している
- 未実装案や作業履歴が混ざっていない
- 関連PRDとADRへの参照がある
- 後続AIが主要境界と依存を理解できる

独立したArchitectureレビュー工程や更新レポートは作らない。高リスク領域は必要に応じて実装レビュー担当へ再確認を依頼する。

完了時は更新ファイル、根拠、不一致、残存リスク、次フェーズの`check-release`を報告する。
