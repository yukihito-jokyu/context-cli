# ADR-002: YAML永続化方式と分割境界

Status: Accepted
Status Reason: 最重レビューで判断境界、選択肢、却下理由、受容リスク、ADR候補との追跡性を確認し、採用を確定した。
Decision ID: ADR-002
Date: 2026-06-07。

## Context

`context` は、ユーザー単位の全体設定と配布先ごとの管理情報をローカルへ永続化する必要がある。初期版の単一ユーザー、単一マシン、ローカル実行に適合し、設定と配布管理情報の責務およびスキーマ変更を分離しながら、安全な更新と互換性検証を実現する方式を実装開始前に固定する必要がある。

この判断は、`docs/engineering/adr-candidates.md`の`ADC-002`を正式化する。

## Decision

`XDG_CONFIG_HOME/context/`配下へ、全体設定を保持する`config.yaml`と配布先ごとの管理情報を保持する`map.yaml`を独立して保存する。`XDG_CONFIG_HOME`が未設定の場合は`~/.config/context/`を使用する。

各ファイルに独立したスキーマバージョンを設け、未知フィールドの拒否、書き込み前検証、排他ロック、同一ディレクトリでの一時ファイル作成と原子的な置換を実施する。複数ファイルの完全なトランザクションは初期版では保証せず、途中失敗時に変更済みと未変更の対象を報告する。

## Options

### Option A: 2つのYAMLファイルへ分割

Pros:

- 設定と配布管理情報の責務およびスキーマ変更を分離できる。
- 利用者が保存内容を確認しやすく、運用上の問題を診断しやすい。
- 初期版の単一ユーザー、単一マシン、ローカル実行に対して実装と運用を単純に保てる。
- ストレージ境界を維持し、要件変更時に代替方式を検討できる。

Cons:

- 複数ファイルをまたぐ完全なトランザクションを単純には保証できない。
- 各ファイルについてスキーマ、検証、ロック、原子的更新を実装する必要がある。

Rejection Reason:

- Not applicable

### Option B: 単一のYAMLファイル

Pros:

- 永続化対象とファイル操作を1ファイルへ集約できる。
- ファイル単位の原子的置換で全データを同時に更新できる。

Cons:

- 全体設定と配布管理情報の責務をファイル単位で分離できない。
- 一方の変更でも共有スキーマとファイル全体へ影響する。
- 配布先情報の増加に伴い、全体設定だけの操作でもファイル全体を扱う必要がある。

Rejection Reason:

- 全体設定と配布管理情報の責務およびスキーマ変更を独立させられず、変更影響を限定しにくいため。

### Option C: 単一のSQLiteデータベース

Pros:

- 複数種類のデータをトランザクションで更新できる。
- データ量や検索条件が増えた場合に構造化された問い合わせを利用できる。

Cons:

- 初期版のデータ量と単一ユーザー運用に対して実装および運用の複雑度が高い。
- 利用者が保存内容を直接確認し、問題を診断する容易さがYAMLより低い。
- SQLiteの依存、スキーマ管理、マイグレーション設計が必要になる。

Rejection Reason:

- 初期版の単一ユーザー、単一マシン、ローカル実行に対して必要以上に複雑であり、保存内容の確認しやすさと最小実装を優先するため。

## Rationale

2つのYAMLファイルへ分割する方式は、初期版の運用条件に適合し、利用者による確認と診断を容易にしながら、全体設定と配布管理情報の責務およびスキーマ変更を分離できる。独立したスキーマバージョン、厳格なデコード、排他ロック、原子的置換を必須とすることで、YAML利用時の互換性と整合性のリスクを抑制する。

## Consequences

- `config.yaml`はContext Repositoryなどの全体設定だけを保持する。
- `map.yaml`は配布先ごとの管理情報だけを保持する。
- 両ファイルは独立したスキーマバージョンを持ち、未対応バージョンでは処理を停止する。
- YAMLの未知フィールドを拒否し、値を検証してから使用または保存する。
- 設定ディレクトリは新規作成時`0700`、設定ファイルは`0600`とする。
- データ量または同時実行要件が変化した場合は、ストレージ境界を維持したまま代替方式をADRで評価する。

## Risks

- 複数ファイル更新の途中で失敗し、変更済みと未変更のファイルが生じる可能性がある。
- YAMLの手動編集により未知フィールド、不正値、未対応スキーマバージョンが混入する可能性がある。
- 同時実行により更新競合やデータ損失が生じる可能性がある。
- 不適切な権限によりユーザー設定が他者から読み取られる可能性がある。

## Mitigations

- 複数ファイルの途中失敗時は処理を停止し、変更済みと未変更の対象を報告する。
- 未知フィールドを拒否し、書き込み前にスキーマと値を検証する。
- 更新時に排他ロックを取得し、取得できない場合は待機せず処理を停止する。
- 同一ディレクトリの一時ファイルへ書き込み、既存ファイルを原子的に置換する。
- 設定ディレクトリと設定ファイルの権限を検証し、既存権限が広すぎる場合は暗黙に変更せず処理を停止する。

## Assumptions

- 初期版は単一ユーザー、単一マシンでのローカル実行を対象とする。
- Context Repositoryと配布先はローカルファイルシステム上に存在する。
- 初期版のデータ量と同時実行要件は、2つのYAMLファイルとファイル単位の排他制御で扱える。
