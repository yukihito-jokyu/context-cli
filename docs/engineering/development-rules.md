# Development Rules

Status: Accepted
Status Reason: 重レビューで依存規則とシンボリックリンク拒否範囲を修正した。永続化の詳細は現状維持とし、必要なADR確定後に再開する。

## Engineering Principles

- 必須: Context Repositoryを唯一の正本とし、配布先の内容から正本を暗黙に更新しない。
- 必須: 既存データとローカル編集を、ユーザーの明示的な許可なく上書きしない。
- 必須: domainをCLI、UI、永続化、OS固有APIから独立させる。
- 原則: 変更範囲をユーザー成果と責務境界に合わせ、機能固有の先行抽象化を避ける。
- 原則: 失敗時は部分的な状態と利用者が取るべき行動を明確にする。
- 推奨: 標準ライブラリと既存依存を優先し、追加依存を最小限にする。

## Implementation Rules

- 必須: 依存方向 `cli → application → domain` を守り、循環依存を作らない。
- 必須: infrastructureはapplicationが定義するI/Oポートとdomainの型にのみ依存し、具象ユースケースには依存しない。
- 必須: CLIコマンド解析と対話UIの外部ライブラリは `internal/cli/` に閉じ込める。
- 必須: YAML、XDGパス、ファイルロック、原子的置換、OSファイルAPIの実装を `internal/infrastructure/` に閉じ込める。
- 必須: 長時間処理と中断可能な処理は `context.Context` を受け取り、キャンセルを伝播する。
- 必須: 中断後に一時ファイルや不完全な管理情報を残さない。
- 必須: 設定ファイルは未知フィールドを拒否してデコードし、値を検証してから使用または保存する。
- 必須: 書き込みは同一ディレクトリの一時ファイルを使用して原子的に置換し、既存ファイルへ直接書き込まない。
- 必須: 配布対象の既存パスがシンボリックリンクの場合は、リンクをたどらず処理を停止する。
- 必須: 配布先までの各パス構成要素がシンボリックリンクの場合は処理を停止する。
- 必須: 管理対象を削除する場合は管理記録で追跡できる対象に限定し、対象一覧を表示して明示確認を得る。
- 原則: 複数ファイル処理の途中失敗時は処理を停止し、変更済みと未変更の対象を報告する。
- 原則: `utils`、`common`、`helpers` のような責務が曖昧なパッケージを作成しない。

## Error Handling

- 必須: 通常の入力エラー、I/Oエラー、競合、キャンセルで `panic` を使用しない。
- 必須: domainとapplicationは、呼び出し側が判定可能なエラーを保ちつつ文脈を付けて返す。
- 必須: CLI境界でエラーを一度だけユーザー向けメッセージと終了コードへ変換する。
- 必須: エラーへ秘密情報、設定内容全体、不要なユーザーパスを含めない。
- 必須: 競合、過剰なファイル権限、未知のスキーマバージョン、ロック取得失敗では暗黙に修復せず停止する。
- 原則: エラーメッセージには失敗した操作、対象、復旧方法を含める。
- 原則: エラーのラップでは `errors.Is` または `errors.As` による判定可能性を維持する。

## Security / Privacy

- 必須: `context` は初期版でネットワークアクセスを行わない。
- 必須: 設定ディレクトリは新規作成時 `0700`、`config.yaml` と `map.yaml` は `0600` で作成する。
- 必須: 既存権限が広すぎる場合は暗黙に権限変更せず、警告して処理を停止する。
- 必須: ファイル更新前に対象が通常ファイルであることを確認し、シンボリックリンクを拒否する。
- 必須: 配布先の識別前に親を含む各パス構成要素を検査し、シンボリックリンクを拒否する。
- 必須: ローカル編集または競合を検出した場合は、明示的な承認を得るまで変更しない。
- 必須: 外部依存の追加時にライセンスと既知のセキュリティ問題を確認する。
- 原則: ログとエラーには必要最小限のパスおよび属性だけを含める。
- Not applicable: 初期版は秘密情報を必要とする外部サービスを使用しない。将来必要になった場合は関連PRDで保管方法を設計する。

## Testing Rules

- 必須: 変更した振る舞い、境界条件、重要な失敗経路に対応するテストを追加または更新する。
- 必須: domainとapplicationのテストは外部I/Oへ依存させない。
- 必須: infrastructureのファイルテストはテスト用一時ディレクトリを使用し、利用者の実設定を変更しない。
- 必須: CLIテストは標準入力、標準出力、標準エラーを差し替え、表示と終了コードを検証する。
- 必須: 主要操作のE2Eテストはネットワークへ接続せず、隔離された一時環境で実行する。
- 原則: Go標準の `testing` とテーブル駆動テストを使用し、アサーションライブラリを追加しない。
- 原則: カバレッジ率を合否の単一基準にせず、未検証の重要分岐をレビューする。
- 推奨: 実装前または実装と同時に、期待する失敗を再現するテストを追加する。
- bootstrap後の必須コマンド: `task test`
- bootstrap後の失敗情報確認コマンド: `task test:errors`
- bootstrap後の必須品質ゲート: `gofmt`、`go vet ./...`、`golangci-lint run`、`govulncheck ./...`、`go test ./...`、文書Lint
- 品質ゲートの実行定義とCIの正本は、`Taskfile.yml`、`.golangci.yml`、`.github/workflows/`とする。

## Code Review Rules

- 必須: mainブランチへ統合する変更はPull Requestでレビューする。
- 必須: レビューでは要件への適合、依存方向、データ損失リスク、競合処理、権限、エラー経路、テストを確認する。
- 必須: 変更理由、検証結果、影響を受ける文書をPull Requestへ記載する。
- 必須: 外部依存の追加または更新では、必要性、保守状況、ライセンス、セキュリティ、バイナリサイズへの影響を確認する。
- 原則: 自動生成物とロックファイルを生成元の変更と一緒にレビューする。
- 原則: 高リスクなファイル更新、削除、マイグレーションは正常系だけでなく中断と部分失敗もレビューする。

## Git / Branch / Commit Rules

- 必須: 変更は短命な作業ブランチで行い、Pull Requestを介して統合する。
- 必須: 1つのPull Requestは1つの目的に絞り、無関係な整形やリファクタリングを含めない。
- 必須: コミットに秘密情報、利用者固有の設定、生成された一時ファイルを含めない。
- 原則: コミットはレビュー可能な変更単位に分ける。
- 推奨: コミットメッセージはConventional Commitsに従う。

## Dependency Management

- 必須: 依存追加前に標準ライブラリまたは既存依存で要件を満たせない理由を確認する。
- 必須: Go依存は `go.mod` と `go.sum`、npm依存は `package.json` と `pnpm-lock.yaml`、Nix依存は `flake.nix` と `flake.lock` で管理する。
- 必須: 直接依存は用途が分かるように宣言し、未使用依存を残さない。
- 必須: 依存追加と更新では、ライセンス、保守状況、既知の脆弱性、互換性をレビューする。
- 原則: Renovateが作成する更新PRで品質ゲートを実行し、major更新は個別に影響を評価する。
- 原則: `minimumReleaseAge` を通常更新へ適用し、脆弱性修正は優先して扱う。
- 推奨: vendoringは行わず、必要性が生じた場合にADRで判断する。

## Configuration / Secrets

- 必須: `config.yaml` と `map.yaml` は `XDG_CONFIG_HOME/context/`、未設定時は `~/.config/context/` に配置する。
- 必須: 両ファイルは独立したスキーマバージョンを持ち、未対応バージョンでは処理を停止する。
- 必須: `config.yaml` は全体設定、`map.yaml` は配布先ごとの管理情報だけを保持する。
- 必須: YAMLの未知フィールドを拒否し、書き込み前にスキーマと値を検証する。
- 必須: 更新時は排他ロックを取得し、取得できない場合は待機せず処理を停止する。
- 必須: 秘密情報をYAML設定へ保存しない。
- 必須: 配布先はシンボリックリンクを解決せず、字句的に正規化した絶対パスを識別子として保存する。
- 原則: 時刻はRFC 3339形式のUTCで保存する。
- Not applicable: 初期版には秘密情報の入力要件がない。

## Logging / Observability

- 必須: 通常のコマンド結果は標準出力、警告、診断、エラーは標準エラーへ出力する。
- 必須: `log/slog` を基盤とし、CLI向けカスタムHandlerでログレベル、色、時刻、属性整形、出力先を制御する。
- 必須: domainとapplicationは `*slog.Logger` へ直接依存せず、必要な場合は利用側で定義した最小インターフェースへ依存する。
- 必須: ログへ秘密情報、設定内容全体、配布物の内容を出力しない。
- 原則: 既定出力は人間向けの簡潔な英語テキストとする。
- 原則: 色は端末能力と非対話実行を考慮して無効化できるようにする。
- 推奨: デバッグ情報は明示的なログレベル指定時だけ出力する。

## Documentation Rules

- 必須: ユーザー向け挙動、設定形式、コマンド、開発規則を変更した場合は関連文書を同じ変更で更新する。
- 必須: Engineering FoundationとAccepted済みADRに反する変更は、先に該当文書の変更フローを実施する。
- 必須: 文書は既存のPrettier、textlint、prh設定で検証する。
- 原則: ユーザー向けCLI出力とエラーメッセージは英語、開発文書は当面日本語とする。
- 原則: コマンド例は実在するコマンドとオプションだけを記載する。
- 推奨: 重要な設計理由はコードコメントへ重複記載せず、ADRまたは設計文書を参照する。

## Definition of Done

- 必須: Acceptance Criteriaを満たし、変更した正常系、境界条件、重要な失敗経路のテストが成功している。
- 必須: `gofmt`、`go vet ./...`、`golangci-lint run`、`govulncheck ./...`、`go test ./...`、文書Lintが成功している。
- 必須: LinuxとmacOSで必要なCIが成功している。
- 必須: 依存方向、エラー処理、権限、競合、シンボリックリンク、中断時の状態を自己レビューしている。
- 必須: ユーザー向け挙動、設定、構造、技術判断に影響がある場合は関連文書を更新している。
- 必須: Pull Requestに変更理由、検証結果、既知の制約、Deferred事項を記載している。
- 必須: 未解決のBlocking事項がない。

## Exceptions

- Not applicable: 現時点で恒常的な例外は認めない。
- 例外が必要な場合は、適用範囲、理由、リスク、期限、承認者を関連PRDの作業記録またはADRへ記録し、人間の承認を得る。

## Related Decisions

- レイヤー構造とYAML永続化は、初回PRD開始前にADRで確定する。
- CLIコマンド解析と対話UIのライブラリは、関連PRDで要件を確定した後のADR候補とする。
- 設定スキーマのマイグレーション、完全な複数ファイルロールバック、公開配布方式は要件発生時の判断候補とする。

## Assumptions

- 品質ゲートを実行するTaskfile、Goモジュール、golangci-lint設定、GitHub Actionsはbootstrap工程で整備済みである。
- 初期版はローカルの単一ユーザー利用であり、チーム共有、リモート同期、秘密情報管理を必要としない。
- CLI出力の国際化要件は初期版にない。

## Open Questions

### Blocking

- レイヤー構造とYAML永続化の判断をAccepted済みADRとして参照できる状態にする。

### Deferred

- Windows向けの権限、ロック、原子的置換、パス規則は、Windows対応を判断するPRDで定義する。
- CLIコマンド解析と対話UIのライブラリは、関連PRDで要件を定義してから選定する。
- CLI出力の国際化方式は、複数言語への対応要件が発生したPRDで決定する。
- JSONログの形式と有効化方法は、機械可読ログの利用要件が発生したPRDで決定する。
- 非対話で削除または上書きを承認するフラグの有無と安全条件は、関連コマンドのPRDで決定する。
- 設定スキーマのマイグレーション手順は、互換性変更が必要になったPRDで決定する。
- 複数ファイル操作の完全なロールバック要件は、`add` または `sync` のPRDで決定する。
