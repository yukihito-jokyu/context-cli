# Technology

Status: Accepted
Status Reason: Cobra + pflagを用いたCLIコマンド定義と、internal配下の独立した非公開パッケージ群によるシンプルな構造への移行を人間が承認した。

## Product Context

`context` は、ローカルのContext Repositoryを正本として、AI開発コンテキストをカレントディレクトリへ安全に配布、再配布、同期する個人開発者向けCLIである。既存データやローカル編集を明示的な許可なく上書きせず、利用者環境への実行時依存を最小化する必要がある。

## Runtime / Platform

- macOSおよびLinux上で動作する単一のGoバイナリを正式対応とする。
- 利用時にNode.js、Nix、その他の外部ランタイムを要求しない。
- 初期版は単一ユーザーによるローカル実行を対象とし、ネットワークアクセスを行わない。

## Languages

- Go: CLI、内部処理、ファイルシステム連携の実装に使用する。
- Goモジュールパスは `github.com/yukihito-jokyu/context-cli` とする。
- Markdown: プロダクト、設計、開発規則、利用方法の文書に使用する。
- Nix: 再現可能な開発環境とツールチェーンの定義に使用する。
- YAML: ユーザー設定、配布管理情報、開発ツール設定に使用する。

## Frameworks / Libraries

- CLIコマンド解析ライブラリは、Cobraおよびpflagを採用する。
- 対話UIライブラリは、`github.com/charmbracelet/huh` を採用する。デモコマンド `prompt-demo` により、単一選択（Select）、複数選択（MultiSelect）、入力検証、およびキャンセル処理の動作を検証済みである。
- `go.yaml.in/yaml/v3`: `config.yaml` と `map.yaml` の読み書きに使用する。
- `log/slog`: ログ基盤に使用し、CLI向けカスタムHandlerでログレベル、色、時刻、属性、出力先を制御する。
- Viperは使用しない。設定の探索、検証、読み書きは明示的に実装する。

## Data Storage

- データベースは使用せず、ユーザー単位のYAMLファイルへ永続化する。
- `config.yaml` はContext Repositoryなどの全体設定を保持する。
- `map.yaml` は配布先ごとの管理情報を単一ファイルで保持する。
- 両ファイルは `XDG_CONFIG_HOME/context/` 配下に置き、`XDG_CONFIG_HOME` が未設定の場合は `~/.config/context/` を使用する。
- 両ファイルは独立したスキーマバージョンを持つ。未対応のスキーマバージョンは自動変換せず、処理を停止する。
- 配布物自体にはバージョンを持たせない。
- 保存する時刻はRFC 3339形式のUTCとする。
- 配布先までの各パス構成要素にシンボリックリンクがないことを確認する。
- 配布先はシンボリックリンクを解決せず、字句的に正規化した絶対パスを識別子として保存する。

## External Services

- Not applicable: 初期版は外部サービスおよびネットワークアクセスを使用しない。
- Context Repositoryのクローンと更新は利用者がCLIの外部で行う。

## Development Tooling

- Nix Flake: Goと開発ツールを含む開発環境の正本とする。
- Go toolchain: ビルド、フォーマット、静的検査、テストに使用する。
- Task: 開発者向けコマンドの統一インターフェースに使用する。
- golangci-lint: Goコードの静的解析に使用する。
- govulncheck: Go本体と依存モジュールにある既知の脆弱性を検査する。
- Delveおよびgopls: デバッグとエディター連携に使用する。
- pnpm、Prettier、textlint、prh: Markdown and設定ファイルのフォーマットおよび文書Lintに使用する。
- Lefthook: コミット前のフォーマットと文書Lintに使用する。
- Renovate: Go、Nix、npmの依存関係更新PRを作成する。

## Testing Stack

- Go標準の `testing` を使用し、アサーションライブラリは導入しない。
- テーブル駆動テストを基本とする。
- `gotestsum` を開発依存として使用し、失敗情報に絞ったテストコマンドを提供する。
- 内部モジュールは一時ディレクトリやモックを使用した単体テスト、CLIコマンドは入出力を差し替えて `Options.Run()` を呼び出す統合テストを基本とする。
- `init`、`add`、`sync` などの主要操作には、ネットワークやユーザーの実設定に依存しないローカルE2Eテストを設ける。

## Deployment / Infrastructure

- 初期版は `go install` とローカルビルドを対象とする。
- GitHub ActionsでLinuxとmacOSのビルドおよびテストを実行する。
- Lintと文書検証はLinuxで実行し、OS固有のファイルおよび権限動作はLinuxとmacOSのマトリクスで検証する。
- GoReleaser、署名済みバイナリ、パッケージマネージャー配布は現時点では採用しない。

## Version Policy

Goのバージョンは `flake.lock` で固定したNix開発環境を正本とし、`go.mod` とCIを同じバージョンへ揃える。依存バージョンは `go.mod`、`go.sum`、`pnpm-lock.yaml`、`flake.lock` で固定し、RenovateのPRでテストと互換性を確認して更新する。外部依存の追加時は、標準ライブラリで代替できない理由、保守状況、ライセンス、既知のセキュリティ問題をレビューする。

## 技術制約事項

- 配布対象は通常ファイルとしてコピーし、シンボリックリンクで接続しない。
- 配布先の既存対象がシンボリックリンクの場合は、リンク先を変更せず処理を停止する。
- 配布先までの各パス構成要素がシンボリックリンクの場合は、同一配布先の重複登録を防ぐため処理を停止する。
- YAMLは未知フィールドを拒否して厳格にデコードし、書き込み前に検証する。
- 設定ディレクトリは新規作成時 `0700`、設定ファイルは `0600` とする。既存権限が広すぎる場合は暗黙に変更せず、警告して処理を停止する。
- 設定ファイル更新時は排他ロックを取得し、取得できない場合は待機せず処理を停止する。
- 書き込みは同一ディレクトリの一時ファイルを使用して原子的に置換し、既存ファイルへ直接書き込まない。
- 複数ファイルの完全なトランザクションは保証しない。途中失敗時は停止し、変更済みと未変更の対象を報告する。
- ローカル編集または競合を検出した場合は停止し、ユーザーの明示的な承認後に限り置換する。
- 管理対象の削除は `context` の管理記録で追跡できる対象に限定し、対象一覧の表示と明示確認を必須とする。

## Related Decisions

- CLIコマンド解析ライブラリはCobraおよびpflagを採用し、対話UIライブラリはhuhを採用する。

## 前提条件

- 利用者はContext Repositoryを事前にローカルへクローンし、利用可能な状態に保つ。
- 初期版は個人開発者による単一ユーザー、単一マシンでの利用を想定する。
- Context Repositoryと配布先はローカルファイルシステム上に存在する。
- Nix、Taskfile、Goモジュール、依存関係、CIはbootstrap工程で整備済みである。

## 未解決事項

### Deferred

- Windows対応は、最初のPRDでパス、権限、原子的置換、ロックの互換性を評価して判断する。
- `log/slog` のJSON出力は、機械可読ログの利用要件が発生したPRDで判断する。
- GoReleaser、署名済みバイナリ、パッケージマネージャー配布は、公開配布要件を定義するフェーズで判断する。
- プロジェクトのライセンスは、外部公開またはバイナリ配布前に決定する。
- 設定スキーマのマイグレーション方式は、互換性変更が必要になったPRDで決定する。
- 複数ファイル操作の完全なロールバックは、`add` または `sync` の要件策定時に必要性を判断する。
