## Story Requirements

### ST-001: 検証済みContext Repositoryを初回設定する

#### Acceptance Criteria

- AC-001: Context Repositoryが未設定の利用者が安全に利用できるローカルRepositoryを指定すると、検証結果と設定対象が表示され、そのRepositoryが以後の配布と同期に使用する設定として保存される。
- AC-002: 初回設定では保存前の承認を求められず、検証に成功した設定対象が保存される。
- AC-003: 初回設定が完了すると、利用者は設定に成功したことと設定されたContext Repositoryを確認できる。

#### Technical Requirements

- TR-001: 初回設定で保存するContext Repositoryは、保存前にST-004で定義する検証をすべて満たさなければならない。
- TR-002: 初回設定は、対応するスキーマバージョンと検証済みのContext Repositoryをユーザー単位の全体設定として保存しなければならない。
- TR-003: 初回設定の書き込みに失敗した場合、未完了の設定ファイルまたは一時ファイルを利用可能な設定として残してはならない。

### ST-002: 確認してContext Repositoryの設定を変更する

#### Acceptance Criteria

- AC-004: 別の安全に利用できるContext Repositoryを指定すると、保存前に現在の設定と変更後の設定対象が表示され、変更を承認するか確認される。
- AC-005: 利用者が変更を承認すると、設定が検証済みの新しいContext Repositoryへ変更され、変更結果を確認できる。
- AC-006: 利用者が変更を拒否した場合、または確認を中断した場合、処理は設定を変更せずに終了し、拒否または中断したことと既存設定が維持されたことを確認できる。

#### Technical Requirements

- TR-004: 設定変更で保存するContext Repositoryは、確認前にST-004で定義する検証をすべて満たさなければならない。
- TR-005: 設定変更は、利用者の明示的な承認を得るまで既存設定を書き換えてはならない。
- TR-006: 設定変更の拒否、確認の中断、または書き込み失敗時に、既存設定の内容を変更してはならない。

### ST-003: 同じContext Repositoryへの再実行を安全に完了する

#### Acceptance Criteria

- AC-007: 設定済みと同じContext Repositoryを再度指定すると、ST-004の検証に成功した場合は設定変更の確認を求められず正常終了する。
- AC-008: 同じContext Repositoryへの再実行では設定が書き込まれず、ST-004の検証の成否にかかわらず既存設定がそのまま維持される。検証に失敗した場合は、問題の対象と理由を確認できる。
- AC-009: 同じContext Repositoryへの再実行が完了すると、設定変更が不要だったことを利用者が確認できる。

#### Technical Requirements

- TR-007: 指定されたContext Repositoryと保存済みContext Repositoryは、シンボリックリンクを解決せず字句的に正規化した絶対パスへ変換し、その文字列が一致する場合に同一と判定しなければならない。
- TR-008: 同一と判定した場合もST-004に従って検証し、検証結果にかかわらず設定変更の確認処理および設定ファイルの書き込み処理を開始してはならない。検証に成功した場合だけ正常終了し、失敗した場合は既存設定を変更せずエラー終了しなければならない。

### ST-004: 安全に利用できないContext Repositoryの設定を拒否する

#### Acceptance Criteria

- AC-010: 指定したパスが存在しない、またはContext Repositoryとして参照できない場合、設定は拒否され、問題の対象と理由を確認できる。
- AC-011: 指定したRepositoryに`projects/`または`utils/skills/`のいずれかが存在しない、`projects/`にプロジェクトが1件も存在しない、各プロジェクトに`skills/`が存在しない、またはSkillに`SKILL.md`が存在しない場合、設定は拒否される。利用者は検出可能なすべての不備について、対象のRepository相対パスと理由を項目別に確認できる。
  <!-- Issue: https://github.com/yukihito-jokyu/context-cli/issues/10 -->
- AC-012: `projects/`、各`projects/<project-name>/skills/`、`utils/skills/`、各Skillディレクトリ、または各Skillの`SKILL.md`を読み取りまたは検索できない場合、設定は拒否される。利用者は読み取れない対象のRepository相対パスと理由を確認できる。
  <!-- Issue: https://github.com/yukihito-jokyu/context-cli/issues/10 -->
- AC-013: Repositoryまたは必須対象を現在の利用者が読み取れない、探索に必要なディレクトリを検索できない、またはグループか他ユーザーに書き込みが許可されている場合、設定は拒否され、権限上の問題を確認できる。
- AC-014: Repositoryへのパスまたは検証対象に許可されないシンボリックリンクが含まれる場合、リンク先を設定対象として扱わず、設定は拒否される。
- AC-015: いずれかの検証に失敗した場合、既存設定があるときはその内容が維持され、初回設定のときは不正な設定が作成されない。

#### Technical Requirements

- TR-009: 設定候補について、パスの存在と種別、次に定める最小固定構造、必須対象の存在および読み取り可能性、現在の利用者による読み取りとディレクトリ検索の可否、グループと他ユーザーへの書き込み許可、パス構成要素と検証対象のシンボリックリンクを保存前に検証しなければならない。
  <!-- Issue: https://github.com/yukihito-jokyu/context-cli/issues/10 -->
- TR-016: Context Repositoryのルートには、通常ディレクトリ`projects/`と`utils/skills/`が存在しなければならない。`projects/`には、空でない単一のディレクトリ名を持つ`projects/<project-name>/`が1件以上存在しなければならない。
- TR-017: 各`projects/<project-name>/`には通常ディレクトリ`skills/`が存在しなければならない。`projects/<project-name>/AGENTS.md`、`CLAUDE.md`、`README.md`は任意の通常ファイルとし、`skills/`は空を許容する。
- TR-018: `utils/skills/`は共通Skillの原本配置先とし、空を許容する。`projects/<project-name>/skills/`と`utils/skills/`の各直下ディレクトリを1つのSkillとして扱い、各Skillには読み取り可能な通常ファイル`SKILL.md`が存在しなければならない。
- TR-019: 必須ディレクトリは実際に列挙できること、必須ファイルは実際に開いて読み取れることを検証しなければならない。Repository自体を探索できない場合を除き、検出可能な構造不備をすべて収集し、ファイル内容または不要な絶対パスを含めず、対象のRepository相対パスと理由を項目別に報告しなければならない。
- TR-010: 検証は、必要な全項目が成功した場合だけ設定候補を保存可能と判定しなければならない。
- TR-011: 検証失敗時は設定ファイルへの書き込みを開始せず、既存設定を変更してはならない。
- TR-012: シンボリックリンクの検証ではリンクをたどって設定候補または必須対象として受け入れてはならない。

### ST-005: 未対応スキーマの既存設定を安全に扱う

#### Acceptance Criteria

- AC-016: 既存設定のスキーマバージョンが未対応の場合、Context Repositoryの初回設定、変更、または再実行の処理は拒否される。
- AC-017: 未対応スキーマにより処理が拒否されると、利用者は互換性がないことと既存設定が変更されていないことを確認できる。
- AC-018: 既存設定に未知フィールドまたは解釈できない値がある場合、設定処理は拒否され、既存設定は維持される。

#### Technical Requirements

- TR-013: 既存設定は、設定処理を開始する前にスキーマバージョン、未知フィールド、および値の妥当性を検証しなければならない。
- TR-014: 未対応スキーマ、未知フィールド、または不正値を検出した場合、自動変換、暗黙の修復、または上書きをしてはならない。
- TR-015: 互換性検証に失敗した場合は、設定候補の検証、変更確認、および設定ファイルの書き込みへ進んではならない。

## Non-Functional Requirements

- NFR-001: 設定処理はmacOSおよびLinux上のローカルファイルシステムで動作し、ネットワークアクセスまたは外部ランタイムを必要としてはならない。
  Applies To: ST-001〜ST-005
- NFR-002: 新規作成する設定ディレクトリは`0700`、設定ファイルは`0600`とし、既存の設定ディレクトリまたは設定ファイルの権限が広すぎる場合は暗黙に変更せず処理を停止しなければならない。
  Applies To: ST-001, ST-002, ST-003, ST-005
- NFR-003: 設定ファイルを更新する処理は排他ロックを取得し、ロック取得後に既存設定を再読込して、初回設定・設定変更の判断に用いた読込時点の状態から変化していないことを検証しなければならない。ロックを取得できない場合、または既存設定が変化していた場合は、待機または書き込みを行わず、既存設定を維持して停止しなければならない。
  Applies To: ST-001, ST-002
- NFR-004: 設定ファイルの更新は同一ディレクトリの一時ファイルを用いて原子的に置換し、既存設定ファイルへ直接書き込んではならない。
  Applies To: ST-001, ST-002
- NFR-005: 通常の処理結果は標準出力へ出力し、警告、診断、エラーは標準エラーへ出力しなければならない。出力から失敗した操作、問題の対象、利用者が取り得る復旧行動を区別できなければならない。
  Applies To: ST-001〜ST-005
- NFR-006: 出力、ログ、エラーには設定内容全体や配布物の内容を含めてはならない。秘密情報または診断に不要なユーザーパスも含めてはならない。
  Applies To: ST-001〜ST-005
- NFR-007: 設定処理の中断または失敗後に、一時ファイル、不完全な設定、または既存設定の部分的な変更を残してはならない。
  Applies To: ST-001, ST-002, ST-004, ST-005
- NFR-008: 性能について、根拠のある数値基準は定義されていない。ローカルの単一ユーザー実行を妨げる不要な待機を設けず、ロック取得失敗時は待機せず停止する。
  Applies To: ST-001〜ST-005
- NFR-009: アクセシビリティは、GUIまたは視覚表現を伴わない初期版のCLIであるため、追加の固有要件はNot applicableとする。
- NFR-010: 法令、契約、および組織ポリシーについて、このPRDに固有の要件はNot applicableとする。
- NFR-011: 可用性について、常駐サービスまたは外部サービスを使用しないローカルCLIであるため、サービス稼働率の要件はNot applicableとする。
- NFR-012: 監視について、常駐プロセスを持たないローカルCLIであるため、外部監視またはメトリクス収集の要件はNot applicableとする。

## Assumptions

- Context Repositoryは利用者が事前にローカルへクローンし、利用可能な状態に保つ。
- 初回設定では保存前の明示確認を必須とせず、検証結果と設定対象を表示して保存する。
- 初期版は個人開発者による単一ユーザー、単一マシンでの利用を対象とする。
- Context Repositoryとユーザー設定はローカルファイルシステム上に存在する。
- `utils/skills/<skill-name>/`は共通Skillの原本配置であり、配布時は各Skillを個別の配布対象として扱う。`utils/skills/`というRepository内の階層自体は配布先へ作成しない。
- `projects/<project-name>/skills/<skill-name>/`はプロジェクト固有Skillの原本配置であり、共通Skillと同様に各Skillを個別の配布対象として扱う。
